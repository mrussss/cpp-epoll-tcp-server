# 任务 2：SIGPIPE 没处理

## 问题说明

Linux 下，如果对端已经关闭连接，服务端继续 `send()`，可能触发 `SIGPIPE`。`SIGPIPE` 默认行为是终止进程，这会导致服务端被单个异常连接打死。

## 修改目标

做两层保护：

1. start 阶段进程级忽略 `SIGPIPE`
2. 所有 `send()` 调用使用 `MSG_NOSIGNAL`

## 修改要求

### 2.1 在服务启动阶段忽略 SIGPIPE

在 `TcpServer::start()` 或项目实际启动服务的位置加入：

```cpp
if (std::signal(SIGPIPE, SIG_IGN) == SIG_ERR)
{
    LOG_INFO("signal SIGPIPE ignore failed");
}
```

如果当前文件没有相关头文件，按需补充：

```cpp
#include <csignal>
```

或者项目已有风格使用 C 头文件，也可以使用：

```cpp
#include <signal.h>
```

保持项目现有风格即可。

### 2.2 send 使用 MSG_NOSIGNAL

把类似代码：

```cpp
send(fd, buffer, size, 0);
```

改成：

```cpp
send(fd, buffer, size, MSG_NOSIGNAL);
```

典型位置可能在：

- `TcpServer::handleWrite`
- 任何直接调用 `send()` 写 socket 的地方

## 禁止事项

不要因为这个任务改：

- 协议结构
- `Response` / `Request` 字段
- epoll 主流程
- worker 线程结构
- `conn_id` 校验逻辑

# 任务 6：错误响应 JSON 格式不合法

## 问题说明

当前项目里可能存在类似：

```cpp
resp.payload = R"({"payload is empty"})";
```

这种 payload 不是合法 JSON object，因为缺少 key-value 结构。

## 修改目标

所有错误响应 payload 必须是合法 JSON 字符串，例如：

```json
{"status":400,"message":"payload is empty"}
```

## 修改要求

统一使用这种格式：

```cpp
R"({"status":400,"message":"payload is empty"})"
R"({"status":400,"message":"payload too large"})"
R"({"status":400,"message":"invalid log format"})"
R"({"status":500,"message":"log write failed"})"
R"({"status":400,"message":"unknown type"})"
```

如果项目里已经有错误响应 helper，可以复用；如果没有，可以在最小范围内新增一个内部 helper，例如：

```cpp
static Response makeErrorResponse(const Request &request, int status, const std::string &message);
```

但要注意：

- 不要改变协议外层编码格式
- 不要改变 response type 的既有语义
- 必须保留 `fd`
- 必须保留 `conn_id`
- 必须保留 `request_id`
- 必须保留 `version`

## 建议实现原则

如果 message 是固定字符串，可以直接使用 raw string literal。

如果 message 是动态字符串，必须避免生成非法 JSON；本轮不要求引入 JSON 库，可以只处理固定错误信息。

## 禁止事项

不要引入：

- `nlohmann/json`

- JsonCpp
- 其他任何第三方依赖

不要修改：

# 任务 9：total_requests 只统计 LOG_PUSH

## 问题说明

如果字段叫 `total_requests`，它应该统计所有成功进入业务层的请求，而不是只统计 `LOG_PUSH`。

当前问题通常表现为：

```cpp
StatsManager::getInstance().incrementRequests();
```

只出现在 `handleLogPush()` 里面。

这会导致：

- `PING` 不计数
- `ECHO` 不计数
- `STATS` 不计数
- unknown type 是否计数不清晰

## 修改目标

把请求统计移动到 `Dispatcher::dispatch()` 入口。

语义定义：

```text
total_requests = 成功 decode 并进入业务层 Dispatcher::dispatch 的请求数
```

## 修改要求

在 `Dispatcher::dispatch(const Request &request)` 的开头增加：

```cpp
StatsManager::getInstance().incrementRequests();
```

然后删除 `handleLogPush()` 里的重复统计。

## 验收标准

每个成功进入 `Dispatcher::dispatch()` 的请求只统计一次。

必须满足：

- `PING` 增加 `total_requests`
- `ECHO` 增加 `total_requests`
- `LOG_PUSH` 增加 `total_requests`
- `STATS` 增加 `total_requests`
- unknown type 如果已经进入 `dispatch`，也按当前语义计入一次

## 禁止事项

不要改：

- `incrementConnections()`
- `decrementConnections()`
- active connection 统计
- log 写入成功/失败统计
- response 协议格式

# 任务 15：Try_pop stop 后不 drain 剩余数据

## 问题说明

当前 `Try_pop()` / `try_pop()` 如果写成：

```cpp
if (data_queue.empty() || _is_stopped)
{
    return false;
}
```

那么一旦 queue 被 stop，即使队列里还有数据，也无法被 drain。

这和阻塞 `pop()` 的常见语义不一致：

```text
队列有数据：先消费
队列为空并且 stopped：退出
```

## 修改目标

`try_pop` 只根据当前队列是否为空决定能不能弹出数据。

## 修改要求

修改为类似：

```cpp
bool try_pop(T &item)
{
    std::lock_guard<std::mutex> lock(mtx_);

    if (queue_.empty())
    {
        return false;
    }

    item = std::move(queue_.front());
    queue_.pop();
    return true;
}
```

如果项目函数名是 `Try_pop`，保持现有命名，不要为了风格统一改 public API。

## 验收标准

必须满足：

- queue 未 stop 且非空：可以弹出
- queue 已 stop 但非空：仍然可以弹出
- queue 为空：返回 false
- 不改变阻塞 `pop()` 的既有语义
- 不改变 `push()` / `stop()` 的接口

## 禁止事项

不要重构整个队列类。

不要改：

- public API 名称
- 模板参数
- worker 调用方式
- request_queue_ / response_queue_ 的整体接口
