## 任务 12：handleAccept 中 client fd 的 epoll_ctl ADD 失败不应退出进程

### 问题说明

如果新 client fd 加入 epoll 失败，这是单个连接的失败，不应该导致整个服务进程退出。

危险写法：

```cpp
if (epoll_ctl(epfd_, EPOLL_CTL_ADD, fd, &event) == -1)
{
    close(epfd_);
    exit(EXIT_FAILURE);
}
```

这会导致：

- 一个异常 client 杀死整个 server
- 错误关闭 `epfd_`
- 服务可用性下降

### 修改目标

当 client fd 的 `EPOLL_CTL_ADD` 失败时：

```text
只关闭当前 client fd
继续 accept 下一个连接
不能 exit
不能 close epfd_
```

### 推荐写法

```cpp
if (epoll_ctl(epfd_, EPOLL_CTL_ADD, client_fd, &event) == -1)
{
    LOG_INFO("epoll_ctl ADD client fd=%d failed, errno=%d", client_fd, errno);
    close(client_fd);
    continue;
}
```

注意顺序：

```text
epoll_ctl ADD 成功之后，才把 fd 放入 connections_
epoll_ctl ADD 成功之后，才 incrementConnections()
```

如果当前代码已经先 `connections_.emplace()`，需要谨慎调整，避免失败路径造成连接统计错误。

### 禁止事项

不要出现：

```cpp
close(epfd_);
exit(EXIT_FAILURE);
abort();
std::terminate();
```

不要因为单个 client fd 失败关闭整个 server。

## 任务 13：handleWrite / drainResponseQueue 中 EPOLL_CTL_MOD 返回值需要检查

### 问题说明

`epoll_ctl(epfd_, EPOLL_CTL_MOD, fd, &event)` 失败如果不检查，连接状态会和 epoll 监听状态不一致，后续可能出现：

- 连接永远无法继续写
- output buffer 卡住
- fd 已关闭但仍被使用
- 难以定位的偶发错误

### 修改目标

封装一个连接事件修改函数，例如：

```cpp
bool TcpServer::modifyConnectionEvents(int fd, uint32_t events);
```

失败时：

```text
记录日志
关闭该连接
返回 false
```

### 推荐实现

在 `TcpServer` 类中新增私有成员函数：

```cpp
bool TcpServer::modifyConnectionEvents(int fd, uint32_t events)
{
    struct epoll_event event;
    std::memset(&event, 0, sizeof(event));
    event.events = events;
    event.data.fd = fd;

    if (epoll_ctl(epfd_, EPOLL_CTL_MOD, fd, &event) == -1)
    {
        LOG_INFO("epoll_ctl MOD failed fd=%d errno=%d", fd, errno);
        closeConnection(fd);
        return false;
    }

    return true;
}
```

如果项目不用 `std::memset`，保持现有风格也可以。

### 替换位置

把这些地方的裸 `epoll_ctl(... EPOLL_CTL_MOD ...)` 替换成 helper：

```cpp
modifyConnectionEvents(fd, EPOLLIN | EPOLLET);
modifyConnectionEvents(fd, EPOLLIN | EPOLLOUT | EPOLLET);
```

典型位置：

- `TcpServer::handleWrite`
- `TcpServer::drainResponseQueue`
- 其他修改 client fd 监听事件的地方

### 注意事项

如果 `modifyConnectionEvents()` 内部调用 `closeConnection(fd)`，那么调用方要避免继续使用已经失效的 `Connection &conn` 引用。

例如：

```cpp
if (!modifyConnectionEvents(fd, EPOLLIN | EPOLLET))
{
    return;
}
```

或者在循环里：

```cpp
if (!modifyConnectionEvents(resp.fd, EPOLLIN | EPOLLOUT | EPOLLET))
{
    continue;
}
```

### 禁止事项

不要修改：

- Dispatcher
- StatsManager
- 协议格式
- worker 线程结构

# 第二阶段：修 7，active_connections 下溢保护

---

## 任务 7：active_connections-- 可能下溢

### 问题说明

如果 `closeConnection(fd)` 对同一个 fd 被调用多次，或者 fd 已经从 `connections_` 删除后又被关闭，就可能重复减少连接数。

危险写法：

```cpp
close(fd);
connections_.erase(fd);
StatsManager::getInstance().decrementConnections();
```

如果没有检查 fd 是否存在，就会导致 `active_connections` 下溢。

### 修改目标

需要两层保护：

```text
1. closeConnection(fd) 必须幂等
2. StatsManager::decrementConnections() 必须有下溢保护
```

---

## 7.1 closeConnection(fd) 必须先查 connections_

### 推荐写法

```cpp
void TcpServer::closeConnection(int fd)
{
    auto it = connections_.find(fd);
    if (it == connections_.end())
    {
        return;
    }

    if (epfd_ != -1)
    {
        epoll_ctl(epfd_, EPOLL_CTL_DEL, fd, nullptr);
    }

    close(fd);
    connections_.erase(it);
    business::StatsManager::getInstance().decrementConnections();
}
```

实际项目里如果命名空间不同，按项目现有风格调整。

### 必须满足

```text
fd 不在 connections_：
- 直接 return
- 不 close(fd)
- 不 decrementConnections()

fd 在 connections_：
- 从 epoll 删除
- close fd
- erase map
- decrementConnections 一次
```

### 注意

如果项目已经因为第 8 项修过 `closeConnection()`，不要重写整个函数，只补必要的幂等保护。

---

## 7.2 StatsManager::decrementConnections() 增加下溢保护

### 不允许写法

```cpp
active_connections--;
```

或者：

```cpp
active_connections.fetch_sub(1);
```

这两种在当前值为 0 时都有下溢风险。

### 推荐写法

如果 `active_connections` 是 `std::atomic<uint64_t>`：

```cpp
void StatsManager::decrementConnections()
{
    uint64_t current = active_connections.load();

    while (current > 0)
    {
        if (active_connections.compare_exchange_weak(current, current - 1))
        {
            return;
        }
    }
}
```

如果项目用其他整数类型，保持同样语义：

```text
当前值 > 0 才减
当前值 == 0 直接 return
```

### 禁止事项

不要改变：

- `incrementConnections()` 的语义
- `total_requests` 统计位置
- `Dispatcher` 逻辑
- `stop()` 主流程
- `conn_id` 校验逻辑

# 第三阶段：修 14，LOG_PUSH JSON key 保守校验

---

## 任务 14：payload.find("level") 会误判 JSON 字段

### 问题说明

危险写法：

```cpp
request.payload.find("level")
```

它不是在检查 JSON key，而是在整个字符串中搜索子串。

例如下面 payload 可能被误判：

```json
{"elevated":"true","service":"x","message":"y"}
```

因为 `elevated` 中包含类似语义子串，但它并不是 `"level"` key。

### 修改目标

本轮只做保守修复：

```text
检查 payload 是否包含明确 JSON key：
"level"
"service"
"message"
```

也就是检查：

```cpp
"\"level\""
"\"service\""
"\"message\""
```

### 推荐写法

```cpp
const bool has_level = request.payload.find("\"level\"") != std::string::npos;
const bool has_service = request.payload.find("\"service\"") != std::string::npos;
const bool has_message = request.payload.find("\"message\"") != std::string::npos;

if (!has_level || !has_service || !has_message)
{
    return makeErrorResponse(request, 400, "invalid log format");
}
```

如果项目没有 `makeErrorResponse`，可以使用当前项目已有的错误响应构造方式，但 payload 必须是合法 JSON：

```cpp
R"({"status":400,"message":"invalid log format"})"
```

### 重要限制

本阶段不要引入 JSON parser。

禁止：

- `nlohmann/json`
- RapidJSON
- JsonCpp
- 修改 `CMakeLists.txt`
- 新增第三方依赖

### 必须保留

不要改变 `LOG_PUSH` 成功路径：

- 日志写入逻辑不变
- 成功响应格式不变
- request 字段保留
- response 字段保留
- `conn_id` 不丢失
- `request_id` 不丢失

### 说明

这是 V4.1 的保守补丁，不是真正 JSON 解析。真正 JSON parser 可以留到 V5 / V6 再做。

# 第四阶段：修 3，worker 异常保护

---

## 任务 3：worker 线程没有异常保护

### 问题说明

worker 线程里如果异常逃出线程入口，会触发 `std::terminate()`，导致整个进程退出。

风险来源包括：

- `Dispatcher::dispatch(req)` 抛异常
- `response_queue_.push(resp)` 抛异常
- 字符串构造抛异常
- 日志格式化抛异常
- 内存分配失败
- 业务 handler 内部异常

### 修改目标

做两层异常防护：

```text
1. worker 线程入口外层 try-catch
2. 每个请求 dispatch 周围局部 try-catch
```

---

## 3.1 外层线程入口保护

推荐结构：

```cpp
workers_.emplace_back([this, worker_id = i]()
{
    try
    {
        business::Dispatcher dispatcher;

        while (true)
        {
            // 原有 worker 主循环逻辑
        }
    }
    catch (const std::exception &e)
    {
        LOG_INFO("Worker %u fatal exception: %s", worker_id, e.what());
    }
    catch (...)
    {
        LOG_INFO("Worker %u fatal unknown exception.", worker_id);
    }
});
```

外层保护用于兜底，防止任何异常逃出线程函数。

---

## 3.2 每个请求 dispatch 周围加局部保护

推荐结构：

```cpp
try
{
    Response resp = dispatcher.dispatch(req);

    if (!response_queue_.push(resp))
    {
        LOG_INFO("Worker %u failed to push response, queue stopped.", worker_id);
        break;
    }
}
catch (const std::exception &e)
{
    LOG_INFO("Worker %u caught exception while handling request: %s",
             worker_id, e.what());

    // 尽量构造 error response 并 push 到 response_queue_
}
catch (...)
{
    LOG_INFO("Worker %u caught unknown exception while handling request.",
             worker_id);

    // 尽量构造 error response 并 push 到 response_queue_
}
```

### 异常响应要求

如果项目结构允许，构造一个 error response 返回给客户端。

必须保留请求身份字段：

```text
resp.fd = req.fd
resp.conn_id = req.conn_id
resp.request_id = req.request_id
resp.version = req.version
```

payload 使用合法 JSON：

```json
{"status":500,"message":"internal server error"}
```

如果项目里 response type 有专门的 ERROR 类型，使用现有 ERROR 类型。

如果项目结构不方便构造 error response，至少必须满足：

```text
异常被记录
worker 不导致整个进程 std::terminate
worker 不无日志吞异常
```

### 不允许的做法

不要这样写：

```cpp
catch (...)
{
}
```

不要无日志吞异常。

不要捕获异常后直接破坏 worker 主循环。

不要因为单个请求异常退出整个进程。

---

## 3.3 保留 fd reuse 防护字段

异常路径尤其要注意保留：

- `req.fd`
- `req.conn_id`
- `req.request_id`

否则可能破坏之前已经修好的 fd reuse 防护。

如果构造 error response 时没有 `conn_id`，`drainResponseQueue()` 可能无法正确判断 stale response。

---

## 3.4 禁止事项

不要改：

- `request_queue_` / `response_queue_` public API
- 队列整体结构
- Dispatcher 对外接口，除非编译必须
- 协议格式
- fd reuse / `conn_id` 逻辑
- `output_buffer` 逻辑
- `stop()` 流程
- `total_requests` 统计位置

不要为了异常保护重构整个线程池。
